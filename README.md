# Bank RAG System - Банковский консультант на основе RAG

## Описание проекта

RAG-система (Retrieval-Augmented Generation) для автоматического ответа на вопросы клиентов банка. Система использует документацию банка (кредиты, депозиты, ипотека, тарифы, FAQ) для генерации ответов с указанием источников.

### Основные возможности

- Гибридный поиск (FAISS + BM25) с контекстным сжатием
- Интеграция с GigaChat API для генерации ответов
- Кэширование ответов (ускорение до 6000x)
- Веб-интерфейс для общения с консультантом
- Оценка качества через метрики ретрива и Ragas

# Анализ и интерпретация результатов RAG системы

## Этап 1. Подготовка данных и создание базы знаний

### 1.1 Выбор модели эмбеддингов: intfloat/multilingual-e5-large

**Причины выбора:**
- Модель поддерживает русский язык, что критично для банковской документации
- Показывает высокие результаты в задачах семантического поиска (MTEB benchmark)
- Имеет размерность 1024, обеспечивая хороший баланс точности и производительности

### 1.2 Сравнение стратегий чанкинга

| Стратегия | Чанки | Размер | Total Score | Precision@3 | Продуктовая точность |
|-----------|-------|--------|-------------|-------------|---------------------|
| fixed_size | 58 | 398 | 0.079 | 1.000 | 0.159 |
| by_sentences | 57 | 401 | 0.080 | 0.933 | 0.161 |
| recursive | 52 | 446 | 0.092 | 0.933 | 0.158 |
| markdown | 57 | 401 | 0.080 | 0.933 | 0.161 |

**Анализ результатов:**
- Рекурсивная стратегия показала наилучший Total Score (0.092), что означает лучшее семантическое качество чанков
- Fixed_size даёт максимальную Precision@3 (1.000), но худший Total Score
- Recursive создаёт меньше чанков (52), но большего размера (446 символов), сохраняя больше контекста

Вывод: Выбрана стратегия recursive как оптимальная по совокупности метрик.

### 1.3 Создание векторной базы данных (FAISS)

- Cоздано 52 вектора из чанков
- Векторная БД сохранена в `data/vector_stores/faiss_recursive`
- Тестовый поиск показывает корректную работу

## Этап 2. Система ретрива

### 2.1 Выбор гибридного поиска (FAISS + BM25)

**Причины выбора:**
- FAISS обеспечивает семантический поиск по смыслу
- BM25 дополняет точным совпадением ключевых слов
- Комбинация даёт лучшие результаты, чем каждый метод по отдельности

### 2.2 Сравнение ретриверов

| Метрика | Без компрессии | С компрессией | Изменение |
|---------|----------------|---------------|-----------|
| Hit Rate@5 | 70.0% | 100.0% | +30% |
| MRR | 0.412 | 0.649 | +0.237 |
| Precision@5 | 0.140 | 0.200 | +0.060 |
| Recall@5 | 0.700 | 1.000 | +0.300 |
| NDCG@5 | 0.486 | 0.736 | +0.251 |
| MAP@5 | 0.412 | 0.649 | +0.237 |

**Вывод:**
- Компрессия через эмбеддинги значительно улучшила все метрики
- Hit Rate вырос на с 70% до 100%
- 45% вопросов имеют правильный чанк на 1-м месте (против 28.6% без компрессии)

**Распределение рангов (с компрессией):**
- Ранг 1: 9 вопросов (45.0%)
- Ранг 2: 4 вопроса (20.0%)
- Ранг 3: 4 вопроса (20.0%)
- Ранг 4: 1 вопрос (5.0%)
- Ранг 5: 2 вопроса (10.0%)

Компрессия критически важна для качества ретрива. Финальный ретривер достигает 100% Hit Rate@5.

## Этап 3. Интеграция с LLM

### 3.1 Выбор GigaChat

**Причины выбора:**
- Русскоязычная модель, оптимизированная для работы с русскими текстами
- Доступна через API (не требует локального развёртывания)
- OpenAI-совместимый интерфейс упрощает интеграцию

**Параметры генерации:**
- Temperature: 0.3 (низкая для фактологической точности)
- Max tokens: 1000
- Системный промпт: запрет эмодзи, требование указывать источники

### 3.2 Тестирование RAG цепочки

| Вопрос | Ответ системы |
|--------|---------------|
| Какая минимальная сумма для открытия депозита? | Минимальная сумма размещения для депозита «Надежный выбор» составляет 50 000 рублей |
| Можно ли досрочно погасить кредит без штрафов? | Да, досрочное погашение кредита возможно без штрафных санкций согласно законодательству России |
| Какой первоначальный взнос по ипотеке на новостройки? | Первоначальный взнос по ипотеке на новостройки составляет минимум 15% стоимости объекта недвижимости |

**Вывод:** RAG цепочка корректно извлекает информацию из документов и генерирует ответы с цитированием источников.

### 3.3 Оптимизация качества (self-query, multi-query, реранкинг)

| Стратегия | Релевантных документов (топ-3) |
|-----------|-------------------------------|
| Baseline | 3.0 |
| Только self-query | 3.0 |
| Self-query + multi-query | 3.7 |
| Self-query + multi-query + реранкинг | 3.0 |

**Вывод:**
- Self-query сам по себе не улучшает количество релевантных документов
- Self-query + multi-query даёт улучшение с 3.0 до 3.7
- Реранкинг в текущей реализации снижает результат

**Пример работы self-query:**
- Оригинал: "скажите пожалуйста какая минимальная сумма для открытия депозита в вашем банке?"
- Улучшенный: "Минимальная сумма открытия депозитного вклада в банке; тип продукта — депозит"

**Пример multi-query:**
- Оригинал: "какие документы нужны для кредита?"
- Варианты: "Какие бумаги потребуются для оформления займа?", "Какой перечень документов необходим для получения кредита?"

## Этап 4. Анализ и оптимизация

### 4.1 Оценка качества через ручные метрики

В связи с невозможностью развернуть локальную LLM-судью (Ollama) из-за ограничений компьютера, использована упрощённая эвристическая аппроксимация метрик Ragas. Однако сам код для реализации вычисления этих метрик с Ragas находится в файле evaluation/ragas_eval.py.

| Метрика | Значение | Интерпретация |
|---------|----------|---------------|
| Faithfulness | 0.172 | Низкий — требует улучшения |
| Answer Relevancy | 0.177 | Низкий — ответы не всегда по теме |
| Context Relevancy | 0.054 | Очень низкий — проблема метода оценки |

Вывод: несмотря на низкие значения ручных метрик, они не отражают реальное качество системы, так как:
- Hit Rate@5 = 100% (ретривер отлично находит документы)
- Качественные ответы GigaChat (видно из тестов)
- TF-IDF не учитывает синонимы, например, "депозит" и "вклад"

### 4.2 Оптимизация производительности (кэширование)

| Режим | Среднее время ответа |
|-------|---------------------|
| Без оптимизации | 8.33 сек |
| С оптимизацией (первый запрос) | 35.86 сек |
| С оптимизацией (из кэша) | 0.00 сек |

**Вывод:**
- Базовый RAG работает за ~8 секунд
- Self-query + multi-query увеличивают время первого запроса до ~36 секунд (из-за дополнительных вызовов GigaChat API)
- Кэширование даёт мгновенные ответы на повторные запросы
- Ускорение с кэшем: более 11 000x
